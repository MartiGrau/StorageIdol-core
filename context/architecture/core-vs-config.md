# Core vs. Configuration vs. Client Graph

Three distinct layers. Confusing them is the most common architectural mistake.

## Layer 1 — Core (shared infrastructure)

**Lives in:** `core/`, `packages/`  
**Changes affect:** every client simultaneously  
**Treat like:** a versioned library release

What belongs here:
- FastAPI webhook handlers (generic — no client logic)
- LangGraph module subgraphs (behavior building blocks, no client-specific routing)
- Third-party SDK wrappers (WhatsApp, ElevenLabs, Twilio, Claude)
- Shared data models (Contact, Conversation, Message, DebtCase)
- Watchdog monitoring sidecar
- Dashboard and backoffice frontends

What does NOT belong here:
- Any reference to a client's CRM URL, credentials, or data schema
- Any hardcoded intent category that is specific to one business domain and not reusable
- Any template message content

## Layer 2 — Client graph (`clients/<id>/graph.py`)

**Lives in:** `clients/<id>/graph.py`  
**Changes affect:** only that client  
**Generated by:** `.claude/agents/provision-client.md` — but it is code, not config

This is where client-specific behavior lives. It wires Core modules into a LangGraph `StateGraph` for a specific client. The wiring choices (which modules, which routing conditions, which escalation triggers) are what differ between clients.

Example: Retras uses `intent/storage` + `debt/soft`. A mortgage servicer uses `intent/financial` + `debt/hard`. They share no graph code, but they share the underlying modules.

## Layer 3 — Configuration (`clients/<id>/config.yaml`)

**Lives in:** `clients/<id>/config.yaml`  
**Changes affect:** only that client  
**Contains:** values, not logic

What belongs here:
- ElevenLabs voice ID
- D+N threshold values and template names
- Escalation contact list
- Knowledge base path
- CRM base URL (non-secret — secrets go in `.env`)
- Which modules are enabled

What does NOT belong here:
- Routing conditions or branching logic (that is the client graph)
- Prompt text (that belongs in module system prompts or knowledge-base/)
- Secrets (those go in `.env` on the client server, never committed)

## Decision guide

| Question | Answer |
|---|---|
| "Should this be shared across all clients?" | Core |
| "Is this a reusable behavior building block?" | Core module |
| "Is this how a specific client wires the modules?" | Client graph |
| "Is this a value that changes per client but is not sensitive?" | config.yaml |
| "Is this a secret or credential?" | `.env` on client server |
| "Is this client-specific CRM integration logic?" | `clients/<id>/mcp/` |
